## 前言

大家好，今天为大家推荐 AI 大神 Andrej Karpathy 的 Claude Code 指南项目，这是一份能让你的 AI 编程助手学会“守规矩”的规则清单，目前在 GitHub 上已获得超过 110k 颗 Star。它的本质是一份不到 70 行的 Markdown 配置文件，专门用来“管教”像 Claude Code、Cursor 这类 AI 编程工具，提高编程效率和正确率。清单提出了四条根本原则：先问清楚再动手、代码能短则短、只改该改的地方，以及把模糊要求变成可验证的目标，它用最小的成本，帮你把AI管教成了真正能提升效率的可靠伙伴。

## 四条原则详解

- 原则一：先问清楚再动手

这是“防瞎猜”条款。AI 最致命的诱惑在于，它总是急于证明自己有用，哪怕在需求模糊不清时，也会用看似自信的代码把缺口填满。Karpathy 曾抱怨，这种时候模型往往会“虚构一个合理但完全错误”的实现。这条原则强制要求 AI：在面对任何模糊描述时，必须先以提问的方式澄清意图、确认边界条件、罗列歧义点，直到人给出明确回答后才开始写第一行代码。本质上，这是在用追问成本替代返工成本——宁可在对话窗口里花三分钟把需求抠清楚，也不要花三十分钟去理解和删改一段精巧但答非所问的代码。

- 原则二：代码能短则短

这条原则对应 AI 过度工程的通病：你让它实现一个工具函数，它可能给你搬出整套设计模式，甚至顺手建了三个新文件。模型常被训练数据中那些“完备解决方案”的权重所驱使，天然倾向把事情复杂化。而“能短则短”本质上是要求它反过来思考——先从最简实现开始，用一个函数能解决的问题绝不上模块，用十行能表达的逻辑绝不铺成五十行。这不仅是审美偏好，更是工程纪律：短的代码表面积小，更容易审查，更少藏污纳垢，也更容易在未来被快速替换。这条原则让 AI 从“擅自加戏的编剧”退回到“只会按指令执行的快枪手”。

- 原则三：只改该改的地方

AI 在修改一个大文件时，常忍不住顺带“优化”沿途看到的旧代码，或是格式化自己碰过的每一行，导致一次本应只改三行的请求，最终产生一个改动遍布全文件的 Pull Request。这种“附带损害”让代码审查变得极其痛苦，也极大增加了引入意外 Bug 的概率。这条原则的厉害之处在于，它要求 AI 模拟一个外科医生：只精准接触病灶，绝不乱摸健康组织。它被明确告知，修改范围必须严格限定在当前任务要求的行和块之内，哪怕是多一个空格的“整洁癖”也不行。这保障了每一次提交都是最小粒度的、可回溯的原子操作。

- 原则四：把模糊要求变成可验证的目标

这是对第一条原则的能力升级——光问清楚还不够，还要在写代码前，把“清楚”变成“可测”。比如一个要求“让按钮更快响应”的任务，会被 AI 拆解为：从点击到状态反馈延迟降至 200ms 以内、移除该路径同步阻塞逻辑、并用一套具体的操作序列来验证。这条原则要求 AI 在动手之前，先输出一份它即将实现的“验收标准”和测试方法，让人类确认“是的，做出这些就算成功”。这相当于在需求层面建立了一个刚性合约，把所有模棱两可的形容词翻译成可观测、可度量的指标，彻底杜绝了开发完成后双方对“是否完成”的认知错位。

这四条原则合在一起，构成了一套完整的行为闭环：用追问把需求固定下来，用可验证目标为结果立下准绳，用最小改动原则约束行动边界，最后用最简代码干净地执行到位。它们没有一条是单纯“技术”上的限制，全是针对“协作心智”的纪律——告诉 AI 如何做一个不多嘴但可靠的搭档。这大概也是它能收获十万星标的原因：它用最少的字，驯住了最多的不确定性。
